# VisualNovel-Interview-RSS 轻量级定向网络RSS聚合生成系统

100% ChatGPT 生成。

一个专为深度游戏玩家、行业研究者设计的优质“视觉小说与开发者访谈”自动化 RSS 聚合订阅系统。

本系统基于 Python 开发，通过全自动的数据管线，将原本散落在各个媒体网站（如 GameSpark 等）且未提供独立订阅源的精彩访谈、深度专栏，精准转化为标准、轻量、无广告的 RSS 订阅源。



## 🌟 项目亮点

* **精准提取**：拒绝粗暴的全文抓取，针对每一个目标站点量身定制解析规则，完美保留头图、作者、精确发布时间与文章标签。
* **真·增量去重**：内置历史指纹库，无论运行多少次，绝不让重复的文章信息轰炸你的阅读器。
* **全自动无人值守**：依托 GitHub Actions，每隔数小时在云端自动运行，无需自备服务器，无需保持电脑开机。
* **对扩展极度友好**：采用模块化设计，无编程基础的进阶用户也能轻松通过微调规则来添加新的网站支持。



## 📐 系统架构与数据流向

即使您不是程序员，也能通过下图轻松理解本系统的工作原理：
```
网页源头 (Website)
    │
    ▼
专属解析器 (Parser) ──> 负责剔除网页广告，只榨取文章标题、图片、链接等核心骨架
    │
    ▼
标准卡片 (Item Model) ──> 将数据化整为零，变成格式统一的“标准文章卡片”
    │
    ▼
增量去重器 (History Engine) <──> 对比 history.json 历史档案：如果是看过的老文章，直接丢弃；新文章才放行
    │
    ▼
RSS 渲染器 (Aggregator) ──> 将新鲜放行的文章打包，输出为标准的 rss.xml 文件
    │
    ▼
您的阅读器 (Feed Reader) ──> 手机/电脑端即时收到精美的新文推送弹窗
```

# 数据流
```
 ┌──────────────┐      抓取原生网页      ┌──────────────┐
 │ Target Site  │ ────────────────────> │    Parser    │ (特定网站解析器)
 │ (目标网站)   │                        └──────────────┘
 └──────────────┘                               │
                                                │ 提取核心信息
                                                ▼
 ┌──────────────┐      加载并比对指纹    ┌──────────────┐
 │ history.json │ <───────────────────> │  Item Model  │ (标准文章卡片)
 │ (历史指纹库)  │   (自动剔除旧文章)     └──────────────┘
 └──────────────┘                               │
                                                │ 仅保留纯新品
                                                ▼
 ┌──────────────┐      拼装转换为XML     ┌──────────────┐
 │  Output RSS  │ <───────────────────> │ Aggregator   │ (主调度聚合器)
 │ (.xml 文件)  │                       └──────────────┘
 └──────────────┘
        │
        │ 托管至 GitHub Pages 或直接读取
        ▼
 ┌──────────────┐
 │ Feed Reader  │ (用户的 RSS 阅读器接收并弹出通知)
 └──────────────┘
```
1. Source（网站源头）： 程序访问预设的网页（如访谈列表页），把整个网页的源代码全部下载下来。

2. Parser（翻译官）： 专为该网站定制的解析器出场。它像一把手术刀，在一堆凌乱的网页代码中，精准地把文章标题、链接、发布时间、封面图挑出来。

3. Item（标准卡片）： 被挑出来的信息会被放进一个统一的“标准文章卡片”盒子里。不管原本的网站长什么样，到了这一步，它们都变成了完全相同的数据格式。

4. Aggregation（传送带聚合）： 所有的卡片被集中到一条传送带上。

5. History（守门员去重）： 守门员拿出一本叫 history.json 的记事本，上面记录着所有以前推送过的文章链接。守门员会对传送带上的卡片逐一检查：如果卡片在记事本上出现过，直接丢弃；如果是全新的文章，就放行，并将它的名字记入记事本。

6. RSS Output（生成订阅文件）： 最终通过检查的所有全新文章卡片，会被打包成一个名为 rss.xml 的标准文件。你的 RSS 阅读器读取这个文件时，就能瞬间刷出最新鲜的访谈内容。

# 各目录作用

models/（数据模型模型库）：
规定了数据在程序内部流转的“法定格式”。它不负责抓取，也不负责保存，只定义一个完整的“文章信息卡片”应该包含哪些字段（如标题、作者、时间等），以此确保所有组件在对接时不会发生格式混乱。

parsers/（网站专属解析器集）：
这里是爬虫规则的聚集地。每一个不同的媒体网站，在这里都会有一个专属的 .py 规则文件。它们各司其职，专门对付自己负责的那个网站的特殊网页结构。

tools/（开发者与维护工具箱）：
这是整个项目保持高可维护性的精髓所在。 它不参与线上的日常抓取，而是供人类维护者在本地使用的工具。当某个网站改版导致抓取失败时，维护者可以使用这里的工具单独测试某个解析器，或者预览生成的 RSS 效果，而不必跑完整个线上流程。

storage/（持久化数据仓库）：
用于存储“记忆”。最核心的文件是 history.json。如果没有它，程序每次运行都会把网站上的老文章当成新文章重新推送一遍，导致用户的阅读器里充斥着重复垃圾信息。

.github/（自动化工厂）：
配置了 GitHub Actions 自动化脚本。通过它，整个项目可以托管在 GitHub 上免费、自动地每隔几小时运行一次，完全脱离本地电脑的限制，实现真正的无人值守自动化运行。


# Item 数据结构

每一篇被抓取到的访谈文章，都会被格式化为如下 8 个非技术维度的基础属性：

site（来源站点）： 标记这篇文章是从哪一个媒体网站抓到的（例如：GameSpark），方便在多源聚合时进行分类筛选。
category（所属栏目）： 文章在原网站属于什么分类（例如：游戏开发者访谈、业界专栏）。
title（文章标题）： 这篇访谈的完整标题文字。
link（原文链接）： 点击后能直接跳转到原网站阅读完整文章的网页绝对地址（URL）。
description（文章摘要/简介）： 访谈的导语或前几百字的缩影，有时会包含内嵌的网页配图。让用户不用点开正文就能快速了解这篇访谈聊了什么。
image_url（特色头图）： 文章的封面图或核心大图的链接。RSS 阅读器会提取它并展现为漂亮的卡片封面。
pub_date（发布时间）： 这篇访谈在原网站公开发表的精确标准时间（精确到分或秒）。
tags（标签组）： 原作者为文章贴上的关键词标签列表（例如：#视觉小说、#Key社、#独立游戏）。

# Parser 解析器系统
### 为什么每个网站设计一个专用 parser？
世界上没有两个网站的网页结构是完全一模一样的。GameSpark 用 `<div>` 标签放标题，另一个网站可能用 `<h2>` 标签。如果试图用一套通用逻辑去解析所有网站，代码会变得极其臃肿且极易出错。“一站一解析器” 实现了策略隔离，改动 A 站的规则绝对不会影响到 B 站。

### parse() 函数返回什么？
它接收目标网页的 HTML 源代码，经过内部的清洗、提取后，返回一个标准的 `Item` 实例列表（即一组标准文章卡片）。

### 项目如何自动发现 parser？
利用 Python 的动态反射与模块导入机制。在 `parsers/__init__.py` 或 `main.py` 中，程序会自动扫描 `parsers/` 目录下所有符合命名规范的文件，并自动实例化它们。这意味着未来你想添加一个新网站的支持，只需要往 `parsers/` 文件夹丢一个新的文件即可，不需要修改主程序的任何一行代码。

### 为什么这样设计？（开闭原则）
这是软件工程中经典的“对扩展开放，对修改关闭”的设计思想。它极大地降低了开源社区协同开发的门槛，新贡献者只需要专注编写自己关心的那个网站的 `parser`，即可完美融入整个系统。

# RSS 生成流程与增量去重
### history.json 为什么存在？
目标网站的列表页通常会保留最近更新的 20 条文章。如果程序每 1 小时运行一次，而这 1 小时内网站只更新了 1 篇新文章，那么其余 19 篇都是老文章。history.json 扮演了“记忆库”的角色，记录了系统自诞生以来抓取过的所有历史文章指纹。

### 如何避免重复推送？
每次抓取到一篇文章，程序会提取其唯一的 link（链接）或计算其内容哈希值（指纹），然后去 history.json 中检索。如果已存在，说明用户在之前的订阅中已经看过了，直接跳过；如果不存在，则判定为“真·新文章”，放行并把这个新链接追加写入 history.json。

### RSS.xml 如何生成？
主程序利用 rfeed 或 feedgen 库创建一个标准的 RSS 框架，填入源名称（如“GameSpark 开发者访谈订阅源”）。然后将通过去重筛选出来的最新一批新文章卡片（以及从历史中保留的最近数十条记录）依次填充进 <item> 标签中，最后输出为标准的 XML 文本文件，并写出到本地磁盘（或直接通过 GitHub Pages 建立网络可达的链接）。

# Tools 调试层深度拆解
tools/ 目录是维持项目长期生命力的核心。网络爬虫最大的痛点在于“网站随时会改版”。一旦改版，爬虫即刻失效。调试层工具专门为了让维护者在几秒钟内定位并修复问题而生：

parser_test.py（单体测试仪）：
解决问题： 无需运行整个繁琐的去重、生成 RSS 的大流程，直接单独运行某一个 parser。
适合场景： 开发新的 Parser，或者当发现某个网站抓不到数据时，快速在本地控制台打印该站的解析结果，检查是网络问题还是代码问题。

parser_dump.py（原始数据抓拍机）：
解决问题： 将目标网站当前的 HTML 源码和解析出的原始 JSON 数据直接抓取并保存为本地文件。
适合场景： 遇到复杂的网页结构或者动态加载内容时，把快照存下来进行静态分析，避免频繁请求网站导致 IP 被封禁。

validate_all.py（全量健康体检）：
解决问题： 遍历并一键运行系统中所有的解析器，检查是否都能正常工作，有没有哪个网站因为改版抛出异常。
适合场景： 项目版本发布前，或者线上运行出现大面积报警时的整体健康排查。

feed_preview.py（订阅源本地预览器）：
解决问题： 将解析出的数据在本地直接渲染成标准的 RSS XML 格式，甚至在本地搭建一个临时小网页供你预览。
适合场景： 检查输出的摘要中 HTML 标签（如图片、换行）在标准阅读器里会不会排版错乱。

stats.py（数据统计分析）：
解决问题： 分析历史数据，统计总共抓取了多少篇文章，各个网站的更新频率如何，生成趋势报表。
适合场景： 运营看板或定期检查系统成果，评估哪些网站已经长期不更新，可以考虑下线。

# GitHub Actions 自动化工厂

### 它会自动做什么？
在 .github/workflows/update_feeds.yml 中配置了定时任务（Cron Job），例如每隔 3 小时自动唤醒一个 GitHub 虚拟服务器。这个服务器会自动下载本项目代码、安装 Python 环境与依赖、运行 main.py 执行全量抓取与去重，并将生成的最新 rss.xml 和更新后的 history.json 自动提交、推送回当前 GitHub 仓库。

### 为什么需要自动检测 parser 是否失效？
网站改版是悄无声息的。如果缺乏自动检测，某个网站两个月前改版了，你的 RSS 只是不再收到新推送，你很难察觉它是“真的没发新文章”还是“爬虫其实已经坏了”。通过 Workflow 可以在解析出错时自动向开发者发送报错邮件或在 GitHub 上提 Issue（当前项目若未完全实现自动通知，通常也是通过 Workflow 运行失败红标来明示），从而做到“早失效，早感知，早修复”。
